First release.

Sends server-side conversion events to the Meta Conversions API from n8n, with the normalization and hashing done to Meta's spec.

### Why

The usual approach is a Crypto node per PII field, a Switch for whether the event has a value, a Code node to build the payload, and an HTTP Request node with the token in a query parameter. Every way that breaks is invisible, because Meta answers `events_received: 1` either way:

- a Crypto node runs whether or not the field has a value, so a missing city is sent as `SHA256("")` — a well-formed hash of nobody
- Meta's rules are specific, and lowercasing everything uniformly is not the same thing
- hashing an already-hashed value silently destroys the match
- a token in the workflow JSON travels with every export

### What it does

- Map raw values; the node normalizes and SHA-256 hashes them to Meta's spec
- Absent fields are omitted rather than hashed empty
- Already-hashed values are detected and passed through, checked **before** normalizing, because a normalizer applied to a digest destroys it
- The token lives in an n8n credential, not the workflow
- Validates before sending, since Meta will not: Event Source URL on website events, event time inside the 7-day window, Value a number or genuinely empty, Purchase carrying a value, and at least one identifier surviving normalization
- Optional batching, off by default — Meta rejects a whole batch if any single event in it is invalid
- Usable as an AI tool

Every normalization rule is unit-tested against the examples in Meta's own documentation. No runtime dependencies. MIT.
